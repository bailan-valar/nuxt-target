## 项目背景

当前系统存在的问题：
1. 账单表（bills）有三个分类相关字段：
   - category_id (UUID) - 分类ID（新增字段，版本15迁移添加）
   - category (string) - 分类名称（旧字段）
   - sub_category (string) - 子分类名称（旧字段）

2. 分类表（categories）修改后，账单表的字段同步不完整：
   - 现有的分类合并功能只更新 category 和 sub_category
   - 没有更新 category_id 字段
   - 缺少分类重命名功能
   - 缺少分类层级调整功能
   - 删除分类时没有检查关联账单

## 用户需求确认

通过问卷收集到的需求：
1. **迁移策略**: 保持三字段同步（不废弃旧字段）
2. **同步操作**:
   - 分类重命名 ✓
   - 分类合并 ✓
   - 分类层级调整 ✓
   - 分类删除时检查关联账单 ✓
3. **历史数据**: 保持现状（不做全量迁移）
4. **前端功能**: 当前已有分类管理界面 (app/pages/finance/categories/index.vue)

## 实现细节
1. **API 端点**: POST /api/yimu-categories/[id]/rename
   - 请求参数: { newName: string }
   - 响应: { success, message, oldName, newName, updatedBills, isParentCategory }

2. **父分类重命名逻辑**:
   - 更新分类表的 name 字段
   - 更新所有 category 字段匹配的账单
   - 注意：category_id 不需要更新（分类 ID 不变）

3. **子分类重命名逻辑**:
   - 更新分类表的 name 字段
   - 更新所有 sub_category 字段匹配的账单（需同时匹配父分类名称）
   - 注意：category_id 不需要更新（分类 ID 不变）

4. **事务处理**:
   - 使用 Prisma $transaction 确保分类更新和账单更新在同一事务中
   - 如果任何操作失败，整个事务回滚

5. **验证逻辑**:
   - 权限验证：只能修改当前用户的分类
   - 重名检查：同一用户、同一收支类型、同一层级下不能有重名分类
   - 状态检查：只能重命名 active 状态的分类
   - 空值检查：新名称不能为空

## 重要发现
**category_id 字段的含义**:
- category_id 是分类表的 UUID，指向 categories.id
- 当分类重命名时，ID 不会改变，只是名称改变
- 因此，重命名操作**不需要更新 category_id**
- 只需要更新 category 和 sub_category 字段（名称字段）

这与 feat-001 (分类合并) 不同：
- 合并操作：账单从源分类迁移到目标分类，category_id 需要更新
- 重命名操作：账单仍属于同一分类，category_id 不需要更新
